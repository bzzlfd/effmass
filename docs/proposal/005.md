# [proposal-005] NCPP 别名系统

## 背景

同一物理量在不同语境下有不同名称。例如：

| 物理量 | UPF 命名 | Vanderbilt 命名 | 其他文献 |
|--------|---------|----------------|---------|
| D/B 矩阵 | `dion` | `B` / `B_ij` | `D` / `D_ij` |
| 局域势 | `vloc` | `V_loc` | `V_local` |
| 径向网格 | `mesh.r` | `radial_grid` | `r_i` |
| 投影子 | `beta` | `β_i` | `projector` |
| 赝波函数 | `chi` / `wfc` | `φ_i` | `pseudo_wfc` |

起*一个*恰当的名字是困难的。
不同背景的读者（UPF 用户 / PSP8 用户 / Vanderbilt 文献读者 / 第一性原理教材读者）看到同一个数据时，可能期望不同的名字。别名系统让每个人都能找到自己熟悉的命名。

## 设计

在 `NCPP` 内部嵌入一个纯引用转发的 `struct alias`，所有别名集中在此：

```cpp
class NCPP {
public:
    // 主数据成员（不变）
    Nonlocal nonlocal;
    PsWFc wfc;
    std::vector<double> vloc;
    // ...

    // 别名系统
    struct Alias {
    private:
        NCPP& p;
        friend class NCPP;
        explicit Alias(NCPP& pp) : p(pp) {}
    public:
        // B 矩阵的别名
        auto& bMatrix() -> DenseMatrix<double>& { return p.nonlocal.b_matrix; }
        auto& dion()    -> DenseMatrix<double>& { return p.nonlocal.b_matrix; }
        auto& B()       -> DenseMatrix<double>& { return p.nonlocal.b_matrix; }
        auto& D()       -> DenseMatrix<double>& { return p.nonlocal.b_matrix; }

        // 局域势的别名
        auto Vloc() -> std::span<double> { return p.vloc; }
        auto& V_local() -> std::vector<double>& { return p.vloc; }

        // 径向网格的别名
        auto& radialGrid() -> std::vector<double>& { return p.mesh.r; }
        auto& r() -> std::vector<double>& { return p.mesh.r; }
        auto& radialGrid() -> std::vector<double>& { return p.mesh.r; }

        // 投影子
        auto& beta() -> std::vector<std::vector<double>>& { return p.nonlocal.beta; }
        auto& projectors() -> std::vector<std::vector<double>>& { return p.nonlocal.beta; }
    };

    Alias alias{*this};  // const: const Alias alias{*this};
};
```

### 用法

```cpp
NCPP ncpp(upf);

// 主名（推荐）
ncpp.nonlocal.b_matrix[0, 0];

// 别名（UPF 用户熟悉的命名）
ncpp.alias.dion()[0, 0];

// 别名（文献读者熟悉的命名）
ncpp.alias.B()[0, 0];
```

### 为什么是成员函数而非 reference 成员

最直观的别名是 reference member——`ncpp.alias.dion` 比 `ncpp.alias.dion()` 更接近"同一个数据的另一个名字"的语义：

```cpp
struct Alias {
    DenseMatrix<double>& dion;   // 直觉的方案
};
```

但 C++ 没有 property 语法，reference member 带来两个实际问题：

1. **初始化列表爆炸**：Alias 包含 10+ 个别名 reference，构造函数必须显式初始化每一个。此后每加一个别名都要改两处（定义 + 初始化列表）。
2. **reference member 删除默认 copy/move**：含有 reference member 的 struct 无法自动生成拷贝/移动赋值运算符。Alias 失效意味着 `NCPP` 的默认拷贝构造函数也跟着失效——为了别名系统牺牲整个类的可复制性，代价过大。

因此用 member function 做引用转发是 C++ 的务实折中：

```cpp
struct Alias {
    NCPP& p;
    auto& dion() { return p.nonlocal.b_matrix; }  // inline，零开销
};
```

只存一个 `NCPP& p`，所有别名是单行转发函数，编译器优化后与 reference 无差，且不影响外层类型的 copy/move。

### 为什么嵌入类内而非自由函数

1. **IDE 补全**：`ncpp.alias.` 后直接列出所有别名，无需记住 namespace 名
2. **作用域隔离**：别名不污染全局/namespace 作用域，不会与标准库名冲突
3. **命名一致**：`ncpp.alias.dion()` 的书写方式与成员访问 `ncpp.vloc` 风格接近
4. **零开销**：纯引用转发，编译器可完全内联

## 方案对比

两个方案——别名放在 NCPP 的名空间内还是名空间外：

**A：alias struct** — `ncpp.alias.dion()`，别名在主名空间之外
**B：直接方法** — `ncpp.dion()`，别名与主 API 平级

### A 的好处

> 别名和主名有清晰的视觉分层。`ncpp.alias.dion()` 传达了一个语义——"这不是必须知道的，这是一个便利入口"。新读者不会被 `dion()`、`B()`、`D()` 分散注意力，可以专注于 canonical name。

> 名空间安全：未来 NCPP 加真正的新方法时，不会被已占用的别名名阻挡。今天加个 `D()` 做对角化，明天加个 `B()` 做带宽——别名名不会抢走核心 API 的命名空间。

> 可被复刻到其他类型：如果以后 `PSP8` 类也想提供别名，可以用同一个模式——`psp8.alias.dion()`。这种一致性比 `psp8.dion()`（有的类型有，有的没有）更干净。

### B 的好处

> 用户来自 UPF 背景，自然地敲 `ncpp.dion()`，这种用法很有诱惑力。

> 通用代码的场景：某些模板函数只需要访问 D 矩阵。`ncpp.nonlocal.D()` 和 `uspp.nonlocal.D()`。

### A → 攻击 B

> "碰巧对齐"就是名空间污染。`ncpp.D()` 看起来是核心 API，但它是 UPF 的遗留命名——其他格式（PSP8）没有 `dion` 这个概念。把它放到 `ncpp.` 一级，等于承诺"这是 NCPP 的稳定接口"，但实际上它只是一个约定别名。

> 未来如果真有一个物理上的 `D` 量需要暴露，B 方案就得选择：要么不叫 `D()`（迁就别名），要么把别名改掉（破坏现有用户习惯）。A 方案没有这个问题——`ncpp.D()` 永远是空的，留给真正的物理量。

> 如果 B 允许现在用 `ncpp.dion()`，一周后再加 `ncpp.alias.dion()`，后者是前者的重复还是替代？分裂的命名入口让文档和示例难以一致。

> 有 IDE 的情况下，`ncpp.alias.` 之后的自动补全可以弥补很多间隙。

> 如果想要通用 D()，更习惯的替代方式是在后面定义重载函数 `D(NCPP)` 和 `D(USPP)`


### 不做什么

- 别名**只有引用转发**，不含任何计算或数据拷贝
- 不做 `#define dion b_matrix` 这类宏别名
- 不在主数据成员上加别名后缀（保持主名干净）
