# [proposal-007] BetaqTables 构造时机：eager（finalize）vs lazy（accessor）

## 背景

`BetaqTables`（以及 `DBetaqTables`、`D2BetaqTables`）是一组从 NCPP 赝势数据构造的径向 Bessel 变换表，用于 H|ψ⟩ 热路径中的 β(q) 插值。当前设计在 `finalize()` 中 **eager** 构造所有表，而 `Callable` / `Gradient::Callable` / `Hessian::Callable` 的构造器只做**验证**（check existence）。

用户提出：能否把构造时机移到 accessor 中，让 Callable 构造器的 `betaqTables(type.z)` 调用既做校验**也做构造**？

## 符号

| 符号 | 含义 |
|------|------|
| `finalize()` | Hamiltonian 后处理阶段（`finalize()` 方法） |
| accessor | `betaqTables()`, `dbetaqTables()`, `d2betaqTables()` |
| Callable ctors | `Callable(parent, ikpt)`, `Gradient::Callable(...)`, `Hessian::Callable(...)` |

## 现状：eager 构造（finalize）

```
finalize()
  ├── PSPFeature 校验（D2Betaq⇒DBetaq⇒Nonlocal）
  ├── sortByL() + diagonalizeNonlocal()     ← NCPP 数据预处理
  ├── BetaqTables / DBetaqTables / D2BetaqTables  ← 全部构造好
  ├── l_max_ / ng_max_
  └── FFT 计划

accessor   → 只读返回（`const` method, `const&` return）
Callable   → 调 accessor 验证存在性
operator() → 裸访问
```

### 关键代码

`finalize()` 中的构造入口（`hamiltonian.cpp:294`）：

```cpp
if (psp_features & NONLOCAL) {
    auto buildTables = [&](ElementData& elem) -> void {
        sortByL(elem.ncpp);
        diagonalizeNonlocal(elem.ncpp);

        elem.betaq_tables.emplace(
            ncpp.mesh.r, ncpp.mesh.rab, mesh_type,
            ncpp.nonlocal.beta, ncpp.nonlocal.angular_momentum);
        elem.betaq_tables->setVolume(omega);

        if (psp_features & DBETAQ)
            elem.dbetaq_tables.emplace(...)->setVolume(omega);
        if (psp_features & D2BETAQ)
            elem.d2betaq_tables.emplace(...)->setVolume(omega);
    };
    for (auto&& t : atom_->eachType()) buildTables(find_elem(t.z));
}
```

accessor 当前是只读的（`hamiltonian.cpp:209`）：

```cpp
auto Hamiltonian::betaqTables(int atomic_number) const -> const BetaqTables& {
    // ... 遍历 elements_ 找到对应 elem
    if (!elem.betaq_tables)
        throw std::runtime_error("not initialised (did you call finalize()?)");
    return *elem.betaq_tables;
}
```

## 提议：lazy 构造（accessor）

```
finalize()
  ├── PSPFeature 校验
  ├── sortByL() + diagonalizeNonlocal()     ← 仍放在 finalize 中
  └── l_max_ / ng_max_ / FFT 计划

accessor   → 首次调用时构造并缓存，后续直接返回
Callable   → 调 accessor → 触发生成
operator() → 裸访问
```

### 关键代码（示意）

```cpp
auto Hamiltonian::betaqTables(int atomic_number) -> const BetaqTables& {
    // ... 遍历 elements_ 找到对应 elem
    if (!elem.betaq_tables) {
        const auto& ncpp = elem.ncpp;
        auto mesh_type = /* 从 ncpp.mesh.type 推断 */;
        elem.betaq_tables.emplace(
            ncpp.mesh.r, ncpp.mesh.rab, mesh_type,
            ncpp.nonlocal.beta, ncpp.nonlocal.angular_momentum);
        elem.betaq_tables->setVolume(canonical_lattice_->volume());
    }
    return *elem.betaq_tables;
}
```

### 前置条件仍保留在 finalize

`sortByL` 和 `diagonalizeNonlocal` 是 NCPP 数据的结构化预处理（将对角化的 D_ij 写入 `ncpp.nonlocal.beta`），它们不依赖具体的表类型，无论 eager 还是 lazy 都需要先执行。所以即使 accessor 变成 lazy，`finalize()` 仍需做：

```cpp
if (psp_features & NONLOCAL) {
    for (auto&& t : atom_->eachType()) {
        auto& elem = find_elem(t.z);
        sortByL(elem.ncpp);
        diagonalizeNonlocal(elem.ncpp);
    }
}
```

## 比较

### 构造时机

| 维度 | Eager（现状） | Lazy（提议） |
|------|-------------|------------|
| BetaqTables 构造 | finalize 中，对所有元素无条件构建 | 首次 `betaqTables(z)` 调用时 |
| DBetaqTables 构造 | finalize 中，设有 DBetaq 标记则对所有元素构建 | 首次 `dbetaqTables(z)` 调用时 |
| D2BetaqTables 构造 | finalize 中，设有 D2Betaq 标记则对所有元素构建 | 首次 `d2betaqTables(z)` 调用时 |
| sortByL+diagNonlocal | finalize 中 | finalize 中（不变） |
| 失败时机 | finalize 阶段 | 首次 H|ψ⟩/grad/hessian 阶段 |

### 浪费分析

| 浪费来源 | Eager | Lazy |
|----------|-------|------|
| 设了 DBetaq 但只用 Callable（不用 gradient） | DBetaqTables 白算 | 不触发 DBetaqTables 构造 |
| 设了 D2Betaq 但只用 Callable | D2BetaqTables 白算 | 不触发 D2BetaqTables 构造 |
| 元素 l_max < 0（无投影子） | BetaqTables 白算 | 不触发构造（Callable 有 l_max guard） |
| 设了 Nonlocal + 所有元素都有非定域投影子 | 无浪费 | 无浪费 |

典型场景对比（Si + O, l_max≥0 for both, DBetaq + D2Betaq 全开，但只用 H|ψ⟩）：

| | Eager | Lazy |
|---|---|---|
| BetaqTables | 2× 构造 | 2× 构造（同，因为 Callable 会触发） |
| DBetaqTables | 2× 构造 | 0× 构造（Gradient 从未实例化） |
| D2BetaqTables | 2× 构造 | 0× 构造（Hessian 从未实例化） |

### const 安全性

| | Eager | Lazy |
|---|---|---|
| accessor 签名 | `const` method | 需要 `mutable`（或变 non-const） |
| `const Hamiltonian&` | 可调 accessor | accessor 触发了隐藏的可变操作 |
| 线程安全 | 天然安全（只读） | 需额外保障（首次构造的 data race） |

这是 lazy 方案**最根本的代价**：要么放弃 accessor 的 const 语义，要么把 `elements_` 标为 `mutable` 并在 accessor 内做 mutable 修改。

### 代码复杂度

| | Eager | Lazy |
|---|---|---|
| finalize() 行数 | ~50 行（含 buildTables lambda + 循环） | ~15 行（仅 sortByL + diagNonlocal） |
| accessor 行数 | ~10 行（查找 + 抛异常） | ~20 行（查找 + 构造 + 抛异常） |
| 表构造逻辑集中度 | ✅ 一个地方 | ❌ 分散在各 accessor 中（重复的 mesh_type 推导、volumn 获取等） |
| 新增表类型（如 D3Betaq） | finalize 中加一个 if 分支 | accessor 中写全套构造逻辑 |

## 总结

```
              │  Eager (现状)        │  Lazy (提议)
──────────────┼──────────────────────┼─────────────────────
  失败时机     │  finalize (尽早)      │  H|ψ⟩ 调用时 (较晚)
  按需构造     │  ❌ 白算无用表        │  ✅ 只用啥算啥
  const 安全   │  ✅ 天然 ok           │  ⚠️ 需要 mutable
  构造逻辑     │  ✅ 集中在 finalize   │  ❌ 分散在各 accessor
  线程安全     │  ✅ 只读              │  ⚠️ 首次构造需同步
  新增表类型   │  ✅ 加一行            │  ⚠️ 写全套构造
```

两者没有绝对优劣，取决于优先考量哪个维度：

- **如果看重「尽早失败 + 集中管理」** → 保持 eager，`finalize()` 作为单一初始化入口
- **如果看重「按需构造 + 不浪费」** → lazy accessor，代价是接受 `mutable` 和构造逻辑分散

当前代码库中 `elements_` 是 `private` 成员，`const` 方法通过 `psp_features_` 和 `l_max_` 已经能推断表的存在性。如果要走 lazy，可行的妥协方案是保持 eager 构造的大框架，但在 `buildTables` 中跳过 `l_max < 0` 的元素——以极低的改动成本消除白算 BetaqTables 的问题，而不触及 const 安全或构造分散化。
